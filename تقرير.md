

# تقرير هندسي شامل: تصميم نظام "مُحسّن أوامر شامل" متعدد الطبقات

## القسم الأول: الملخص التنفيذي

البحث الميداني في الفترة بين أواخر 2024 وأوائل 2026 أنتج ثلاثة محاور تقنية ناضجة يمكن البناء عليها بشكل إنتاجي مباشر:

أولاً، تحسين المطالبات لم يعد عملاً يدوياً اعتمادياً، بل أصبح خط أنابيب قابل للأتمتة عبر أُطر مثل
**DSPy / MIPROv2**
و
**Promptomatix**
و
**PRewrite**
بمعدل تحسن موثق يتراوح بين 20-60% في الدقة على معايير قياسية، و30% تحديداً وفق تجربة Anthropic Console.

ثانياً، حماية أوامر النظام من التناقض الذاتي والاختراق أصبحت لها بنية رسمية تُسمى
**Instruction Hierarchy**
وضعتها OpenAI ورُسّخت في الـ frontier models، مع أدوات داعمة مثل
**StruQ**
و
**Signed-Prompt**
و
**NeMo Guardrails**.

ثالثاً، استخراج المطالبات المضمنة في الكود انتقل من تحليل الـ AST التقليدي إلى استخدام
**Tree-sitter**
الذي يدعم 66 لغة برمجية، مع دمج محرك التحقق الدلالي (TypeScript Type Checker، Python ast، Go AST) لضمان عدم كسر الكود عند إعادة الكتابة.

**الحكم الجوهري**: التطبيق المطلوب مُمكن إنتاجياً اليوم، لكنه يتطلب فصل صارم بين ثلاثة محركات معزولة، يربطها مُوجِّه دلالي (Semantic Router) مركزي، مع طبقة تقييم آلية مستقلة قبل الإصدار.

## القسم الثاني: الهيكلية المعمارية الكلية (System Architecture)

### 2.1 المخطط العلوي (Top-Level Flow)

التطبيق يتكون من سبع طبقات مرتبة هرمياً:

| الطبقة | الاسم التقني | المسؤولية الجوهرية |
|--------|-------------|------------------|
| 1 | **Ingestion Gateway** | استقبال المدخلات من ثلاثة قنوات منفصلة (Web UI، CLI، IDE Extension) مع تطبيع التشفير |
| 2 | **Input Type Classifier** | تصنيف المدخل إلى واحد من ثلاثة: `user_prompt`، `system_prompt`، `code_artifact` |
| 3 | **Semantic Router** | توجيه الطلب إلى المحرك المتخصص بناءً على التصنيف وميزات إضافية (لغة، طول، تعقيد) |
| 4 | **Specialized Engines** | ثلاثة محركات معزولة، كل منها له ذاكرة وقواعد تحسين خاصة |
| 5 | **LLM-as-a-Judge Layer** | تقييم آلي للمخرج قبل الإصدار، مع رفض وإعادة المحاولة عند الفشل |
| 6 | **Output Validator** | فحص نهائي للسلامة (PII، حقن، تسريب) |
| 7 | **Audit + Telemetry** | تسجيل كامل للقرارات والمقاييس لتحسين النموذج لاحقاً |

### 2.2 تدفق البيانات (Data Flow)

التدفق الكامل من الاستلام إلى الإصدار يمر بالخطوات التالية:

```
Input → Sanitization → Classification → Routing → Engine Processing 
→ Quality Gate (LLM-as-Judge) → Safety Validation → Cache Update → Response
```

**نقطة جوهرية**: كل محرك متخصص له خط أنابيب فرعي خاص به، لكنهم يتشاركون نفس طبقة التقييم النهائية لضمان توحيد معايير الجودة.

### 2.3 آلية التوجيه الدلالي (Semantic Routing)

التوجيه ليس مبنياً على قواعد بسيطة (regex)، بل على نموذج تصنيف خفيف يعتمد على:

**BERT Embeddings**

عبر مكتبة
**Rust Candle**
لتوليد متجه دلالي (384-dim)، ثم تصنيف بشبكة خطية مدربة على ثلاث فئات. هذا النهج هو نفسه المستخدم في
**vLLM Semantic Router**
الإنتاجي مع زمن استجابة دون 10ms. الميزات المضافة للتصنيف تشمل:

- وجود علامات XML أو JSON كثيفة (مؤشر system prompt)
- وجود تعريفات دوال أو import statements (مؤشر code)
- نسبة الفعل الأمري إلى مجموع الكلمات (مؤشر user prompt)

### 2.4 طبقات التخزين المؤقت (Caching Architecture)

البحث في إنتاج LLM في 2026 يُجمع على بنية ثلاثية الطبقات:

| الطبقة | التقنية | معدل التوفير | الاستخدام |
|--------|---------|------------|-----------|
| L1 | Exact-Match Cache | 100% (تخطي LLM) | تكرارات دقيقة |
| L2 | Semantic Cache (Vector Similarity) | 100% (تخطي LLM) | استعلامات معاد صياغتها |
| L3 | Prompt Prefix Cache (provider-side) | 50-90% خصم | تشارك السياق الثابت |

دراسة CoreOps AI (يناير 2026) أثبتت أن
**Semantic Cache**
وحده يحقق معدل إصابة 67% في أعباء الإنتاج، بشرط استخدام آلية
**QuerySignature**
ذات خمس مستويات للمطابقة.

## القسم الثالث: الطبقة الأولى - تحسين أوامر المستخدم النهائي

### 3.1 الإطار العام

الطبقة الأولى تواجه أصعب مشكلة: المستخدم النهائي عادةً يقول "حسّن هذا" دون تعريف معنى التحسين. الحل المعتمد اليوم هو نمط
**RCCF**
(Role-Context-Constraint-Format) الذي يفرض على المُحسِّن استخراج هذه العناصر الأربعة من الطلب الخام.

### 3.2 خط الأنابيب الداخلي للطبقة الأولى

| المرحلة | التقنية | الإخراج |
|---------|--------|---------|
| 1. Intent Extraction | LLM-based classifier with rule-based fallback | بنية JSON تحتوي: `task_type`، `domain`، `success_criteria` |
| 2. Slot Filling | Few-shot prompting with structured output | ملء الفجوات: الجمهور، النبرة، القيود، التنسيق |
| 3. Strategy Selection | Decision tree على نوع المهمة | اختيار بين CoT، Few-shot، Decomposition، Self-Critique |
| 4. Prompt Synthesis | Meta-prompt template | المطالبة المُحسَّنة الأولية |
| 5. Self-Refinement Loop | Critique-Revise pattern | تحسن إضافي 10-25% (موثق) |
| 6. Validation | LLM-as-a-Judge | إصدار أو إعادة |

### 3.3 خوارزميات Intent Extraction

البحث الحالي يميز بين ثلاث طرق:

**Hybrid Classifier**

براءة اختراع 2025 (USPTO 12406660) تصف منهجاً هجيناً يدمج:
- Rule-based classifier للأنماط القياسية
- LLM-based classifier للحالات الحدية

دقتها الموثقة 79% في استخراج النية من تعليقات مراجعة الكود، وفق ورقة "Intention is All You Need" (فبراير 2025).

**Slot Extraction with Semantic Enrichment**

يُغذي الـ LLM ببروميت يحوي عبارات نموذجية تشترك في علاقة دلالية، ثم يستخرج النية من النص الجديد ويُضيفها للذاكرة. مناسب للأنظمة التي تتعلم تدريجياً.

**Pseudo-Answer Generation**

تقنية إعادة كتابة الاستعلام (موثقة في Elasticsearch Labs، يناير 2026): توليد إجابة وهمية أولاً، ثم استخراج المطالبة المحسنة من الإجابة المتوقعة. فعالة جداً للاستعلامات الإعلامية.

### 3.4 قوالب Meta-Prompting

البحث في 2026 يؤكد أن
**Meta-Prompting**
يعمل فقط مع قالب صارم. القالب الموصى به:

```
أنت خبير هندسة مطالبات. استخرج من الطلب التالي:
1. النية الأساسية (task_type)
2. الجمهور المستهدف
3. القيود الصريحة والضمنية
4. تنسيق الإخراج المطلوب
5. معايير النجاح القابلة للقياس

ثم أعد بناء المطالبة وفق نمط RCCF.
أرجع المطالبة النهائية فقط بدون تفسير.
```

نقطة حرجة من ورقة Salesforce (Promptomatix): يجب توليد بيانات تدريب اصطناعية على طول خط الأنابيب لاختيار الاستراتيجية المثلى لكل نوع مهمة.

### 3.5 الميزانية الحوسبية المتوقعة

المهام البسيطة (تحسين بروميت قصير): استدعاء واحد إلى نموذج خفيف (Haiku أو ما يكافئه) بتكلفة دون 0.001 دولار.

المهام المعقدة (مطالبة طويلة بعدة قيود): 3-5 استدعاءات (Extract → Synthesize → Critique → Revise → Judge) بتكلفة 0.01-0.03 دولار.

## القسم الرابع: الطبقة الثانية - هندسة أوامر النظام

### 4.1 جوهر المشكلة

أوامر النظام الإنتاجية اليوم تتراوح بين 2,000 و15,000 كلمة. المشاكل المزمنة فيها أربعة:

| المشكلة | المظهر | المعدل في الإنتاج |
|---------|--------|-------------------|
| Internal Contradictions | تعليمات تنفي بعضها البعض | شائع جداً في البروميتات الموروثة |
| Instruction Drift | تعليمات قديمة لم تُزَل بعد تغيير الميزات | 60%+ من البروميتات الإنتاجية |
| Jailbreak Vulnerability | فجوات تسمح بتجاوز القيود | معدل اختراق ابتدائي 67.21% بدون حماية |
| Hierarchy Confusion | عدم وضوح أولوية التعليمات | يؤدي لسلوك غير متوقع عند التضارب |

### 4.2 البنية المعيارية لأمر نظام فعّال

الإجماع الحالي (Anthropic + OpenAI + ميادين Frontier) على ثمانية مكونات بترتيب صارم:

| الترتيب | المكون | الغرض |
|---------|--------|-------|
| 1 | **Identity Block** | تعريف الهوية والاسم والإصدار |
| 2 | **Capabilities Block** | ما يستطيع النموذج فعله |
| 3 | **Constraints Block** | الحدود الصارمة (instruction hierarchy: highest priority) |
| 4 | **Tone & Style Block** | نبرة الصوت والأسلوب |
| 5 | **Output Format Block** | بنية الإخراج المتوقعة |
| 6 | **Tool Usage Block** | متى يستدعي أدوات وكيف |
| 7 | **Safety Guardrails** | معالجة المحتوى الحساس |
| 8 | **Examples Block** | 3-5 أمثلة مغطية لتوزيع المدخلات |

### 4.3 آلية كشف التناقضات الداخلية

البحث الحالي يقدم منهجين موثقين:

**Self-Consistency Audit**

استخدام نموذج LLM ثانٍ (judge) لقراءة بروميت النظام كاملاً وإصدار قائمة بكل أزواج التعليمات المتناقضة. الكفاءة موثقة بدقة 90% مقارنة بالمراجعين البشريين (ورقة Boğaziçi University، 2025).

**Constraint Graph Analysis**

تحليل البروميت كرسم بياني للقيود (constraint graph) حيث كل عقدة تعليمة وكل ضلع علاقة (يدعم/يناقض/مستقل). الخوارزمية ترصد:
- Cycles بين القيود (تناقض دائري)
- Contradictory Edges بين عقدتين
- Unreachable Nodes (تعليمات ميتة)

### 4.4 منع الاختراق (Jailbreak Prevention)

المرجع المحوري: ورقة OpenAI "The Instruction Hierarchy" (Wallace et al., 2024) التي رفعت مقاومة الاختراق بنسبة 63%. التقنيات المثبتة في الإنتاج:

| التقنية | المنشأ | فعالية موثقة |
|---------|--------|-------------|
| **Instruction Hierarchy** | OpenAI 2024 | +63% مقاومة |
| **Self-Reminder** | NeurIPS 2024 | تخفيض الاختراق من 67.21% إلى 19.34% |
| **StruQ (Structured Queries)** | Berkeley 2024 | فصل صارم بين instruction وdata |
| **Signed-Prompt** | 2024 | توقيع رقمي للتعليمات الموثوقة |
| **Delimiter Hardening** | Industry Standard | استخدام `<system>` و`<user>` بصرامة |
| **Input Size Limits** | OWASP 2026 | حد أقصى للمدخل لتقليل سطح الهجوم |

**ملاحظة جوهرية لمايو 2026**: حادثة GTG-1002 (نوفمبر 2025) أثبتت أن الاختراق يمكن أن يُؤطَر كاختبار دفاعي مشروع. الحماية الحديثة تتطلب:
- فحص intent قبل المعالجة (وليس بعدها)
- multi-layer guardrails متعاقبة
- Constitutional AI rules مدمجة مباشرة في البروميت

### 4.5 خط أنابيب الطبقة الثانية

| المرحلة | الإجراء | الأداة |
|---------|---------|--------|
| 1. Parse | تحليل البروميت إلى مكوناته الثمانية | LLM with structured output |
| 2. Audit | كشف التناقضات والـ drift | Constraint Graph Analyzer |
| 3. Hierarchy Validation | فحص أن القيود في أعلى الهرم | Rule-based + LLM judge |
| 4. Adversarial Testing | اختبار 50+ نمط جيلبريك معروف | JailbreakBench + EasyJailbreak |
| 5. Reconstruction | إعادة بناء البروميت بهيكلية معيارية | Template engine |
| 6. Diff Reporting | تقرير التغييرات للمستخدم | Custom diff renderer |

## القسم الخامس: الطبقة الثالثة - معالجة المطالبات داخل الكود

### 5.1 جوهر التحدي الهندسي

هذه الطبقة الأصعب تقنياً لثلاثة أسباب:

أولاً، المطالبات في الكود تُكتب بطرق متعددة:
- f-strings في Python
- Template literals في JS/TS
- Concatenated strings عبر متغيرات
- Heredocs في YAML/JSON externalized
- Multi-line strings مع تنسيقات معقدة

ثانياً، تحريرها يجب ألا يكسر:
- بنية الـ string interpolation
- المتغيرات المُمررة (placeholders)
- منطق الكود المحيط
- اختبارات الوحدة المعتمدة على المطالبات

ثالثاً، كل لغة لها قواعدها الخاصة، فحل لغوي واحد لا يصلح للجميع.

### 5.2 التقنية المختارة - Tree-sitter

المسح البحثي يُجمع على أن
**Tree-sitter**
هو الحل الإنتاجي الموحد:

| الميزة | القيمة |
|--------|--------|
| اللغات المدعومة | 66 لغة (Python، JS/TS، Go، Rust، Java، إلخ) |
| سرعة التحليل | sub-millisecond per file |
| المنفعة | concrete syntax tree بدلاً من AST مجرد |
| التكامل | MCP-compatible عبر مشاريع جاهزة |
| القدرة | parsing incrementally على تعديلات الكود |

البديل لكل لغة على حدة (Python `ast` module، TypeScript Compiler API، Babel parser) يبقى متاحاً، لكن Tree-sitter يقدم API موحد.

### 5.3 خوارزمية اكتشاف المطالبات في الكود

الإجراء الموثق يتكون من ست مراحل:

**المرحلة 1: Parsing**

تحليل الملف عبر
**Tree-sitter**
للحصول على CST (Concrete Syntax Tree).

**المرحلة 2: String Node Collection**

جمع كل عقد النصوص التي تتجاوز عتبة طول معينة (مثلاً: 80 حرف).

**المرحلة 3: Heuristic Filtering**

تصفية أولية بناءً على إشارات دلالية:
- وجود كلمات مفتاحية ("you are"، "respond"، "assistant"، "instructions")
- وجود placeholders (`{variable}` أو `${var}`)
- موقع الـ string (داخل استدعاء API لـ LLM)

**المرحلة 4: Contextual Classification**

استخدام LLM خفيف لتصنيف كل مرشح: prompt / not prompt. الكفاءة الموثقة على CodeBench: 91% precision.

**المرحلة 5: Variable Tracking**

تتبع أي placeholders داخل المطالبة لضمان عدم كسرها أثناء التحسين. هذا يستدعي ربط الـ string node بسياقه (function args، class members، imports).

**المرحلة 6: Safe Rewriting**

إعادة كتابة المطالبة مع الحفاظ على:
- جميع الـ placeholders بنفس الأسماء
- التنسيق المضاعف (escape characters)
- الـ indentation للملف
- بنية الـ string (single/double quotes)

### 5.4 آلية الحفاظ على Code Integrity

النهج الإنتاجي يعتمد على ثلاث طبقات تحقق:

| الطبقة | الفحص | الأداة |
|--------|-------|-------|
| 1. Syntax | الكود يُحلل بنفس الـ parser بعد التعديل | Tree-sitter re-parse |
| 2. Type | لا تكسر للأنواع (TS/Python type hints) | TypeScript Type Checker / mypy |
| 3. Behavioral | اختبارات الوحدة لا تزال تمر | pytest / vitest في sandbox |

ميزة مهمة: مقالة Satellytes (موثقة) تُذكّر أن الـ AST وحده لا يكفي للأنواع المعقدة. يجب استخدام **TypeScript Type Checker API** (`getTypeOfSymbolAtLocation`) للحصول على الدلالة الكاملة.

### 5.5 استراتيجية التطبيق (Diff Strategy)

النهج المتبع في Cursor و Aider الإنتاجيين:
1. توليد الـ patch كـ unified diff
2. تطبيقه في sandbox معزول
3. تشغيل linter + type checker + tests
4. إذا فشل أي منها: rollback تلقائي مع تقرير للمستخدم
5. إذا نجح: تقديم الـ patch للمراجعة البشرية

### 5.6 الأدوات الإنتاجية المرشحة

|--------|-----------|---------|
| **tree-sitter** | محرك التحليل | MIT |

## القسم السابع: نماذج التقييم (Evaluation Metrics)

### 7.1 إطار التقييم متعدد المستويات

التقييم يجب أن يحدث في ثلاث نقاط:

| النقطة | النوع | الأسلوب |
|--------|-------|--------|
| Pre-deployment | Offline batch eval | DeepEval + golden dataset |
| Pre-response | Online quality gate | LLM-as-a-Judge fast model |
| Post-deployment | Production telemetry | A/B testing + user feedback |

### 7.2 المقاييس الأساسية لكل طبقة

**الطبقة الأولى (User Prompts)**:

| المقياس | التعريف | الأداة |
|---------|---------|-------|
| **Intent Preservation** | هل النية الأصلية محفوظة | LLM judge with rubric |
| **Specificity Gain** | زيادة الدقة في التعليمات | Token-level structural analysis |
| **Output Quality Delta** | فرق جودة الإخراج (مع/بدون تحسين) | Side-by-side comparison + judge |
| **Token Efficiency** | نسبة المعلومة لكل توكن | Custom metric |

**الطبقة الثانية (System Prompts)**:

| المقياس | التعريف | الأداة |
|---------|---------|-------|
| **Contradiction Score** | عدد التناقضات الداخلية | Constraint graph + judge |
| **Jailbreak Resistance** | معدل صد الهجمات | JailbreakBench (200+ patterns) |
| **Instruction Adherence** | اتباع التعليمات تحت ضغط | HarmBench |
| **Hierarchy Compliance** | احترام أولوية التعليمات | OpenAI IH-Eval |

**الطبقة الثالثة (Code Prompts)**:

| المقياس | التعريف | الأداة |
|---------|---------|-------|
| **Syntax Preservation** | الكود يبقى صحيحاً | Re-parse via tree-sitter |
| **Test Pass Rate** | اختبارات الوحدة لا تزال تمر | Sandbox execution |
| **Type Safety** | لا تكسر للأنواع | TypeScript Compiler / mypy |
| **Behavioral Equivalence** | السلوك الإجمالي محفوظ | Golden output comparison |

### 7.3 معالجة تحيزات LLM-as-a-Judge

البحث في 2025-2026 رصد ست تحيزات يجب التحييد ضدها:

| التحيز | الوصف | الحل |
|--------|-------|------|
| **Position Bias** | تفضيل الموضع الأول/الأخير | Randomize order + run both |
| **Verbosity Bias** | تفضيل الإجابات الأطول | Penalize length في الـ rubric |
| **Self-Preference** | تفضيل مخرجات نفس عائلة النموذج | Use cross-family judges |
| **Rubric Order Bias** | حساسية لترتيب المعايير | Multiple rubric permutations |
| **Score ID Bias** | حساسية لأرقام التقييم | Use letter grades بدلاً من أرقام |
| **Reference Answer Bias** | تأثر بالإجابة المرجعية | Provide multiple references |

النهج المعتمد:
**G-Eval**
يولد سلسلة CoT للمعايير أولاً، ثم يستخدمها كخطوات تقييم. كفاءته 90% مقارنة بالمراجعين البشريين.

### 7.4 معدلات النجاح المستهدفة

من المسح المرجعي للأنظمة الإنتاجية المماثلة:

| الطبقة | المقياس | المعدل المستهدف |
|--------|---------|----------------|
| الأولى | Intent Preservation | ≥ 95% |
| الأولى | Output Quality Improvement | ≥ 25% (vs original) |
| الثانية | Jailbreak Resistance | ≥ 80% (vs baseline 33%) |
| الثانية | Contradiction Detection | ≥ 90% recall |
| الثالثة | Syntax Preservation | 100% (hard requirement) |
| الثالثة | Test Pass Rate | ≥ 99% |

## القسم الثامن: الحالات الحدية وحلولها (Edge Cases)

### 8.1 الطبقة الأولى - حالات حدية

| الحالة | المشكلة | الحل المقترح |
|-------|---------|-------------|
| **مطالبة قصيرة جداً** ("ساعدني") | لا توجد نية واضحة | تفعيل وضع المحادثة (clarifying dialogue) قبل التحسين |
| **مطالبة بلغات مختلطة** | فشل التصنيف | Language detection أولاً، ثم routing لمحرك متخصص بكل لغة |
| **طلبات متناقضة داخلياً** | مستحيل تحسينها | إرجاع تحذير + تقديم خيارات بديلة |
| **مطالبات تتضمن أسرار** (API keys) | مخاطر تسريب | PII/secrets scanner قبل المعالجة |
| **طلبات إبداعية متعمدة الغموض** | "اكتب قصيدة غامضة" | استثناء من نمط RCCF + احترام النية الأصلية |

### 8.2 الطبقة الثانية - حالات حدية

| الحالة | المشكلة | الحل المقترح |
|-------|---------|-------------|
| **بروميت طويل جداً** (>20K كلمة) | تجاوز نافذة السياق | تجزئة ذكية حسب المكونات + معالجة قطعة قطعة |
| **بروميتات مدمجة** (system in system) | بنية متداخلة | Recursive parsing + treat each level independently |
| **أوامر بلغات متعددة** | حوكمة غير متسقة | Normalize إلى لغة واحدة (مع caveat) |
| **اعتمادات على tools خارجية** | الأمر يفترض أدوات لم تُذكر | Tool inventory check قبل التقييم |
| **بروميتات Constitutional AI** | تحوي قيم متضاربة بطبيعتها | حالة خاصة: لا توصف كتناقضات |

### 8.3 الطبقة الثالثة - حالات حدية

| الحالة | المشكلة | الحل المقترح |
|-------|---------|-------------|
| **مطالبات مولدة ديناميكياً** (concatenated) | الـ string الكامل يتكون runtime | Static analysis + trace باستخدام symbolic execution |
| **مطالبات مخزنة خارجياً** (YAML/JSON files) | ليست في الكود مباشرة | فحص متعدد المصادر + cross-reference |
| **مطالبات في تعليقات** (docstrings) | قد تكون مهمة أو لا | Heuristic + بشرية optional |
| **اللغات النادرة** (COBOL، Erlang) | لا يدعمها Tree-sitter | Fallback إلى regex-based detection مع تحذير |
| **كود ضخم** (>100K أسطر) | تكلفة المعالجة | Incremental indexing + watch mode |
| **مطالبات تستخدم meta-syntax** (Jinja داخل F-string) | تداخل معالجة | Specialized parser لكل تركيبة |

### 8.4 الحالات المشتركة (عبر الطبقات)

| الحالة | الحل |
|-------|------|
| **انقطاع شبكة LLM** | Circuit breaker + cached responses + queue retry |
| **تكلفة محسوبة قبل التنفيذ** | Token budget estimation + user approval threshold |
| **تعارض الإصدارات** | Versioned prompts + rollback capability |
| **GDPR/خصوصية** | PII redaction + user-owned data + deletion API |

## خاتمة

التطبيق المقترح، رغم تعقيده، قابل للتنفيذ بشكل كامل بالأدوات المتوفرة في مايو 2026. المفاتيح الجوهرية لنجاحه ثلاثة:

أولاً، **الفصل الصارم** بين الطبقات الثلاث - كل طبقة لها بنية معالجة مستقلة وتُقيَّم بمعايير خاصة بها.

ثانياً، **الالتزام بـ Instruction Hierarchy** كمبدأ معماري وليس مجرد تقنية - في تصميم البروميتات الداخلية للتطبيق نفسه.

ثالثاً، **التقييم المستمر** كميزة مدمجة من اليوم الأول، وليس كإضافة لاحقة - المنظومات التي تربح هي تلك التي تُقاس وتتحسن آلياً عبر DSPy + MIPROv2 وما يماثلها.

الأبحاث الأحدث (Promptomatix من Salesforce، يوليو 2025) تشير إلى أن السوق يتجه نحو "ديمقراطية" تحسين البروميتات - أي أن النظام الذي تبنيه يجب أن يخدم غير الخبراء بنفس فعالية الخبراء، وهذا هو الميزة التنافسية الحقيقية لـ "مُحسّن أوامر شامل".
